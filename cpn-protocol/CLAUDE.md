# CLAUDE.md — Протокол CPN (Cerberus)

## Статус

🟢 **Реализован.** Полностью разработан и задокументирован протокол CPN (Code Name: Cerberus) — защищенный транспортный протокол с многослойной маскировкой трафика и гибридной пост-квантовой криптографией.

## Обзор

CPN — это фундаментальный протокол безопасности, разработанный для создания защищенного VPN-туннеля, устойчивого к глубокому анализу пакетов и блокировкам. Протокол реализован на языке Rust с использованием асинхронного runtime Tokio.

## Ключевые особенности

### 1. Гибридная пост-квантовая криптография
- **X25519 ECDH** — классический эллиптический обмен ключами
- **Kyber-768 KEM** — пост-квантовая инкапсуляция ключей
- **Комбинирование:** `HKDF-SHA512(X25519 || Kyber-768)`
- **Forward Secrecy:** гарантирована для каждой сессии

### 2. Многослойная маскировка транспорта
- **Первичный (TLS):** Эмуляция TLS 1.3 (Chrome 132) с фейковым ClientHello
- **Вторичный (WebSocket):** Трафик через CDN (Cloudflare) с маскировкой под чат
- **Третичный (QUIC):** UDP с эмуляцией QUIC Initial Packet
- **Автоматическое переключение:** при 3 таймаутах подряд

### 3. Защита от анализа трафика
- **Паддинг:** 0-255 байт случайного заполнения
- **Таиминг-обфускация:** джиттер 3-18 мс
- **Keep-alive:** 1-4 секунды (рандомно)
- **Уникальные профили:** индивидуальные параметры для каждого клиента

### 4. Сессионное шифрование
- **Алгоритм:** AES-256-GCM
- **Nonce:** 12 байт (4 случайных + 8 счетчика)
- **Смена ключа:** каждые 80 секунд или 120 МБ
- **Replay-защита:** кольцевой буфер 1024 пакета (O(1))

## Транспортные уровни

### 1. TCP + Эмуляция TLS 1.3

**Особенности:**
- Полностью идентичный Chrome 132 ClientHello
- Набор шифров, расширения, порядок байт — копия реальных дампов Wireshark
- SNI: microsoft.com, cloudflare.com, yandex.ru (обновляется раз в 24 часа)
- После рукопожатия — переход в режим CPN

**Структура:**
```
ClientHello → ServerHello → ChangeCipherSpec → Finished
    ↓
[CPN Handshake: X25519 + Kyber-768 в session_ticket]
    ↓
[Зашифрованный трафик AES-256-GCM поверх "TLS-туннеля"]
```

### 2. WebSocket через CDN

**Активация:** При 3 таймаутах TCP-транспорта

**Особенности:**
- Подключение к домену за Cloudflare
- Заголовки идентичны десктопному Chrome
- Опкод BINARY, но с JSON-префиксом `{"t":"message","d":`
- Размер фреймов: 64-4096 байт (случайный)

### 3. UDP + Эмуляция QUIC

**Особенности:**
- Первый пакет — полная копия QUIC Initial от Chrome 132
- Версия: 0x00000001, Connection ID: 8 случайных байт
- В Token вшивается зашифрованный ключ сессии CPN
- После обмена — переход на чистый CPN поверх UDP

**Применение:** Низкая задержка (игры, VoIP)

## Криптографическое ядро

### Обмен ключами

```rust
// Генерация эфемерных ключей
let x25519_secret = X25519Secret::generate();
let kyber_secret = Kyber768Secret::generate();

// Комбинирование
let combined = concat(
    x25519_secret.diffie_hellman(&peer_x25519_public),
    kyber_secret.decapsulate(&kyber_ciphertext)
);

// Мастер-ключ
let master_key = hkdf_sha512(combined, salt, info);
```

### Сессионное шифрование

```rust
struct CpnPacket {
    nonce: [u8; 12],        // 4 случайных + 8 счетчика
    sequence: u64,          // Номер пакета
    ciphertext: Vec<u8>,    // Зашифрованные данные
    tag: [u8; 16],          // Тег GCM
}
```

**Смена ключа:**
- Время: 80 секунд
- Объем: 120 МБ
- Процесс: Новый обмен X25519 + Kyber внутри канала

### Защита от replay-атак

```rust
struct ReplayWindow {
    bits: [AtomicU64; 16],  // 1024 бита
    lower: AtomicU64,       // Нижняя граница
}
```

- O(1) на все операции
- Lock-free реализация
- Автоматическое расширение окна

## Управление соединениями

### Entry Server

**Роли:**
- Прием входящих соединений (TCP:443, UDP:443)
- Аутентификация по прешареному ключу (Argon2id)
- Туннелирование к Exit Server
- Replay-защита для каждого клиента

### Exit Server

**Роли:**
- Проксирование во внешний интернет
- NAT traversal
- Не хранит ключи клиентов
- Не видит реальные IP

### Control Server

**Функции:**
- Выдача профилей маскировки
- Обновление списков SNI и Split Tunneling
- Мониторинг доступности нод
- REST API (порт 9090/9091)

## Split Tunneling

**Реализация:**
- Список российских доменов (Госуслуги, Сбер, VK, Яндекс и т.д.)
- IP-диапазоны обновляются раз в 12 часов
- Трафик к RU-доменам идет мимо туннеля
- Правила маршрутизации через nftables/iptables

## Профили маскировки

**Структура:**
```json
{
  "padding_shift": 12,
  "sni_list": ["microsoft.com", ...],
  "keepalive_interval": 2.5,
  "jitter_profile": "normal",
  "transport_priority": ["tls", "quic", "websocket"]
}
```

**Особенности:**
- Уникальны для каждого клиента
- Генерируются Control Server'ом
- Обновляются раз в 24 часа
- Исключают сигнатурный детект

## API

### Основные эндпоинты

- `GET /api/status` — статус сервера
- `GET /api/stats` — статистика
- `GET /api/clients` — список клиентов
- `POST /api/transport/switch` — смена транспорта
- `GET /api/profile/{id}` — профиль маскировки
- `GET /api/sni/{region}` — список SNI
- `GET /api/split-tunnel` — правила split tunneling

### WebSocket API

Реальное время:
- Смена транспорта
- Новые подключения
- Статус клиентов
- Ошибки

## Безопасность

### Достигнуто

✅ **Конфиденциальность:** AES-256-GCM, ключи в RAM  
✅ **Целостность:** GCM-тег проверяется до обработки  
✅ **Аутентичность:** прешареный ключ + подписанные сертификаты  
✅ **Forward Secrecy:** гарантирована  
✅ **Пост-квантовая стойкость:** Kyber-768 в гибриде  
✅ **Невозможность сигнатурного детекта:** уникальные профили + паддинг + джиттер  
✅ **Защита от replay:** кольцевой буфер  

### Анализ безопасности

**Преимущества:**
- Гибридная криптография (классика + пост-квантовая)
- Регулярная смена ключей
- Side-channel защита
- Lock-free структуры данных

**Потенциальные уязвимости:**
- Side-channel атаки (время выполнения)
- Генерация случайных чисел
- Утечка памяти
- Теоретические атаки на Kyber

## Технологический стек

- **Язык:** Rust 1.78+
- **Async:** Tokio

- **TUN:** tun2
- **WebSocket:** tokio-tungstenite
- **TLS-парсинг:** ручная сборка по RFC 8446
- **Веб-панель:** warp + svelte
