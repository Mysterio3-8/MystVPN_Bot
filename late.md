## 🔴 Архитектурный реверк (триггер: 50+ платящих)

### Multi-Protocol Adaptive Key (MPAK) — единый ключ со всеми транспортами
**Триггер:** 50+ платящих + нужна надёжность под маркетинг.
**Что:**
- Один JSON-конфиг со всеми outbound'ами (Reality, XHTTP, WebSocket+TLS, Shadowsocks)
- Observatory каждые 30 сек проверяет живость каждого
- Балансировщик автоматически выбирает канал с минимальным пингом
- Юзер просто включает VPN — всё остальное само
- Routing-правила: UDP-звонки → XHTTP, РФ-сайты → direct, остальное → balancer

**Код:** методы в XrayService:
- `generate_adaptive_key(user_id)` — собрать JSON с текущими параметрами
- `_build_reality_outbound()`, `_build_xhttp_outbound()`, `_build_ws_outbound()`

### Android MAX-защита (стелс-режим)
**Триггер:** Жалобы Android-юзеров на перебои / система их режет.
**Что:**
- DNS-over-HTTPS вместо системного DNS

- Reality SNI на `stun.l.google.com` (маскировка под голосовой звонок)
- strict_route: true (весь трафик принудительно в туннель)

**Код:** вариант конфига в `_build_key(platform="android_secure")`

## 🟠 Анти-цензурные эшелоны (триггер: 50–500 платящих)

> Цель этого блока: работать при любом сценарии блокировок — от текущего ТСПУ до полного Чебурнета.
> Внедрять послойно: сначала 1й эшелон, потом 2й и т.д.

---

### Эшелон 1: XHTTP — починить резервный канал (ближайший)
**Триггер:** 50+ платящих ИЛИ первые жалобы что Reality нестабилен.
**Проблема:** Сейчас XHTTP сломан — `serverName: "77.110.96.77"` (IP вместо домена), сертификат не валидируется.
**Минимальный фикс (1ч):** В 3x-ui inbound XHTTP заменить serverName на `keybest.cc`. Проверить порт 2052 в ufw.
**Зачем важно:** XHTTP работает поверх HTTP/3 (QUIC = UDP) — это и резервный канал, и решение для WhatsApp/Telegram-звонков (UDP трафик идёт через него без потерь).

**Код после фикса:** в `_create_client_once` добавить в XHTTP-настройки `serverName: "keybest.cc"`.

---

### Эшелон 2: WhatsApp/Telegram звонки — server-side routing
**Триггер:** Массовые жалобы после первой маркетинговой волны.
**Что:** В 3x-ui создать outbound `direct` + routing rule для UDP-портов:
- WhatsApp: 3478-3479, 5060-5061, 16384-16414 (STUN/TURN/RTP)
- Telegram: 50000-60000 (UDP MTProto)
- Rule: если протокол UDP и порт в этом диапазоне → outbound direct (не через туннель)

**Важно:** Это server-side решение. Клиентский Hiddify Bypass RU/CIS для split-tunneling РФ-сервисов — отдельная вещь (инструкция в site/pages/guide).

---

### Эшелон 3: Russian Relay VPS (обход белых списков)
**Триггер:** Жалобы что VPN не работает в корпоративных сетях / конкретных провайдерах.
**Идея:** Цепочка Клиент → Российский VPS (Yandex Cloud / VK Cloud / Timeweb) → Зарубежный сервер.
Российские IP входят в белые списки по умолчанию. Трафик между RU-VPS и зарубежным сервером — VLESS Reality (устойчивый к анализу).

**Что нужно:**
1. Арендовать RU-VPS: ~500-1000 ₽/мес (Yandex Cloud compute, VK Cloud, Timeweb)
2. Создать inbound на RU-VPS на 443 с маскировочной HTML-страницей
3. Relay: VLESS → RU-VPS → VLESS Reality → основной сервер
4. В боте: отдельная подписка "Белые списки" / "Корпоративный"

**Дополнительно:** Путь в 3x-ui замаскировать под реальный API:
- Примеры path: `/api/v1/portal/updates`, `/static/js/app.js`, `/api/v2/notifications`
- Сервер на 443 должен отвечать валидным HTML на прямые GET-запросы (сбивает автосканеры)

**Маркетинговый смысл:** Тариф "Бизнес" / "Проф" дороже на 100-150 ₽/мес, для тех кто работает через корп-сеть.

---

### Эшелон 4: WebSocket через CDN (Cloudflare/российский CDN)
**Триггер:** Провайдеры начали блокировать прямые IP нашего сервера.
**Что:** Трафик идёт через CDN (Cloudflare Workers или российский CDN). DPI видит только CDN-IP которые никогда не заблокируют (на них сидят банки, госуслуги и т.д.).
- Cloudflare Workers бесплатно до 100k реквестов/день
- Трафик выглядит как HTTPS к `cloudflare.com`
- WebSocket поверх HTTPS с JSON-префиксом: `{"t":"message","d":...}` (имитация чата)
- Размер фреймов: 64-4096 байт (случайный)

**Важно:** Cloudflare может заблокировать в РФ — тогда нужен российский CDN (Qrator, Stormwall, CDN-service.ru).

---

### Эшелон 5: Режим осады (Чебурнет-сценарий)
**Триггер:** РКН заблокировал всё кроме IP Сбербанка и Госуслуг.
**Это крайний сценарий. Реализовывать не раньше 500+ платящих.**

**Что:**
1. **Yggdrasil Mesh-сеть** — децентрализованная оверлейная IPv6-сеть. Работает через NAT, Wi-Fi, Bluetooth без центральных серверов. Если хотя бы у одного соседа/узла есть выход — все участники сети получают связность. Для звонков и чата хватает даже при полном блоке интернета.
   - Реализация: portable Yggdrasil-клиент в ZIP-архиве, бот присылает при нажатии "Аварийный выход"
   - Каждый юзер = узел сети → чем больше юзеров, тем надёжнее сеть

2. **WebRTC как транспорт** — видеозвонки РКН не заблокирует (парализуют госорганы). WebRTC (STUN/TURN) = P2P-протокол для звонков. XRay умеет работать поверх WebRTC как транспорт. Задержка высокая (100-300 мс), но для текста/ссылок подойдёт.

3. **"Режим осады" в боте** — при активации бот присылает три конфига на выбор:
   - Стандартный (Reality TCP) — для обычных дней
   - Белые списки (WS + Russian relay) — для корп-сетей и ненадёжных провайдеров
   - Чебурнет (Yggdrasil + инструкция) — на крайний случай

**Код бота:** в `handlers/cabinet.py` кнопка "🛡️ Режим осады" → три конфига → пресылает нужный.

### Phantom Noise Generator — идея активной маскировки
**Статус:** Идея задокументирована. Текущая реализация в ТЗ технически некорректна.

**Идея:** Генерировать тысячи фейковых TLS-сессий чтобы перегрузить DPI и заставить его пропускать реальный трафик в "слепой зоне". Маркетингово — "активная защита от РКН".

**Почему текущий ТЗ не работает:**
DPI стоит между российским юзером и нашим сервером. Phantom Generator в ТЗ запускается на нашем сервере и подключается к нашему же серверу (порт 8443). Эти соединения идут внутри дата-центра и вообще не проходят через DPI. Перегрузки нет.

**Что нужно чтобы это работало:**
Фейковые соединения должны идти из России к серверу — т.е. с тысяч российских IP-адресов. Это требует либо:
1. Партнёрства с российскими пользователями (добровольная распределённая сеть — юзеры запускают tiny-клиент и шумят для всей сети)
2. Арендованных российских прокси/residential IP — дорого, ~$100+/мес за нормальный пул
3. Собственной mesh-сети (см. Yggdrasil выше) — юзеры автоматически становятся нодами

**Маркетинговый потенциал:** Если сделать правильно — уникальное УТП в РФ-рынке. "Вы не только защищены, вы часть распределённой сети которая делает цензуру неэффективной."

**Тариф "Фантом":** 349 ₽/мес (стандарт 219 ₽). Дополнительные фичи: приоритет при перегрузке, активная маскировка. Реалистично при 1000+ юзеров когда есть сеть нод.

**Триггер для реализации:** 1000+ платящих + массовые жалобы на блокировки + бюджет.
