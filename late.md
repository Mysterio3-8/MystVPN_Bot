## 🔴 Архитектурный реверк (2-3 недели Opus-кода)

### Multi-Protocol Adaptive Key (MPAK) — единый ключ со всеми транспортами
**Триггер:** Когда наберём 50+ платящих и нужна надёжность под маркетинг.
**Что:** 
- Один JSON-конфиг со всеми outbound'ами (Reality, XHTTP, WebSocket+TLS, Shadowsocks)
- Observatory каждые 30 сек проверяет живость каждого
- Балансировщик автоматически выбирает канал с минимальным пингом
- Юзер просто включает VPN — остальное сами себя разруливает
- routing-правила: UDP-звонки на XHTTP, российские сайты напрямую, остальное на балансер

**Код:** методы в XrayService:
- `generate_adaptive_key(user_id)` — собрать JSON с текущими параметрами
- `_build_reality_outbound()`, `_build_xhttp_outbound()`, `_build_ws_outbound()` — каждый транспорт отдельно

**Почему отложено:** Сейчас Reality работает достаточно. Нужен trigger реальной проблемы (блокировка у юзеров).

### Android MAX-защита (стелс-режим)
**Триггер:** Жалобы от Android-юзеров на перебои / система их режет.
**Что:**
- DNS-over-HTTPS вместо системного DNS
- TUN-интерфейс без SOCKS5 на localhost (чтобы MAX не видел подозрительные порты)
- Reality SNI на `stun.l.google.com` (маскировка под голосовой звонок)
- strict_route: true (принудительно весь трафик в туннель)

**Код:** вариант конфига в `_build_key(platform="android_secure")`

**Почему отложено:** Сейчас Android-юзеров мало, нет реальных жалоб.

### Ротация Reality-ключей по расписанию
**Триггер:** Первые сигнатурные детекты (юзеры жалуются что Reality перестал работать неделю-две).
**Что:** Крон каждые 48 часов меняет publicKey, shortId, SNI в 3x-ui. Старые клиенты работают 24ч, новые получают обновлённый ключ.

**Код:** задача в Celery/APScheduler + метод в XrayService.rotate_reality_keys()

**Почему отложено:** Пока нет признаков детекта.

### mTPROXY для Telegram API (защита бота от блокировки IP)
**Триггер:** Бот упал один раз потому что IP попал в бан Telegram.
**Что:** Отдельный mTPROXY на второй VPS или хостинге. Бот ходит через него в Telegram API.

**Почему отложено:** Риск низкий, юзеров мало.

### Anti-fraud для триала (IP + fingerprint проверка)
**Триггер:** Видно что люди создают 10 аккаунтов в день с одного IP для бесплатного триала.
**Что:** В UserService при create_user проверка: не больше 3 регистраций с одного IP за неделю, не больше 1 регистрации с одного fingerprint (если webapp) за месяц.

**Почему отложено:** Пока это не проблема (всего 2 платящих).

### Бэкап конфигов в Redis
**Триггер:** 3x-ui панель упала, потеряли все ключи.
**Что:** При create_client пишем в Redis hset всё что нужно для восстановления. Скрипт восстановления читает Redis и пересоздаёт через API.

**Почему отложено:** Пока один основной сервер, риск падения низкий.

### Health-check мониторинг + автоалерты
**Триггер:** Нужно понимать что сервер упал до того как юзеры начали жаловаться.
**Что:** Внешний мониторинг (с другого VPS в РФ) каждые 5 минут проверяет живость сервера. При падении — alert в TG админу.

**Почему отложено:** Пока живое тестирование показывает что всё работает.

## 🟠 Технические улучшения VPN

### XHTTP резервный канал — сделать рабочим
**Триггер возврата:** Когда Reality начнёт блокироваться у юзеров (массовые жалобы) ИЛИ когда наберём 100+ платящих.
**Что:** Сейчас XHTTP сломан — `serverName: "77.110.96.77"` (IP вместо домена), сертификат не валидируется.
**Минимальный фикс:** В 3x-ui inbound XHTTP заменить serverName на `keybest.cc`. Проверить порт 2052 открыт в ufw.
**Полный фикс:** Описан в чате 2026-05-02 (multi-outbound с balancer + observatory).
**Зачем не сейчас:** На Reality всё работает, юзер 2 человека — резервный канал не критичен.

### Multi-protocol адаптивный конфиг (balancer + observatory)
**Триггер:** 500+ платящих + первые сигнатурные детекты Reality.
**Что:** Один ключ — два outbound (Reality + XHTTP), Observatory ping каждые 30 сек, balancer leastPing.
**Почему отложено:** Каждые 30 сек ping → лишний трафик и точки отказа. Observatory багает на iOS-клиентах. Преждевременная сложность.

### Свой сервер + .ru домен под белые списки
**Триггер:** Корпоративные клиенты или сети с агрессивными белыми списками жалуются на блокировки.
**Что:** Арендовать второй VPS, привязать к домену типа `cdn-update.ru`/`api-service.ru` с маскировочной статической страницей. WebSocket+TLS на 443 с путём `/api/v1/updates`.
**Почему отложено:** Недели работы + деньги (домен ~$10/год + VPS ~$5-10/мес + Let's Encrypt + статика). Преждевременно для 2 платящих.

### TUN-режим со sniffing для звонков
**Триггер:** Жалобы на качество WhatsApp/Telegram звонков через VPN.
**Что:** Включить TUN-режим в клиентском приложении (Hiddify/v2rayTUN уже умеют), sniffing для автодетекта QUIC.
**Почему отложено:** На клиентских приложениях это уже работает по умолчанию. Свой код не нужен.

### Health-rotation на серверной стороне
**Триггер:** 1000+ платящих + сигнатурный детект.
**Что:** Watchdog который при детекте быстро переключает inbound на резервный с другим SNI/портом.
**Почему отложено:** В памяти `project_rotation_safety.md` — пересоздание inbound автоматом = риск дубль-рассылок. Только ручное.

## 🟡 Продуктовые фичи

### Тариф «Бизнес» с приоритетным UDP
**Триггер:** Когда базовый тариф работает у 95% юзеров без жалоб.
**Что:** Премиум тариф с гарантированным качеством звонков, отдельный inbound с QoS.
**Почему отложено:** Сложно, ноль ценности при 2 платящих.

### Свой мобильный клиент (iOS + Android)
**Триггер:** 100+ платящих + понимание что Hiddify/v2rayTUN — слабое место воронки (юзеры путаются).
**Что:** См. `vpn-app/CLAUDE.md`. Нативный клиент с логином через TG.
**Почему отложено:** App Store review 2-4 недели, разработка минимум 1-2 месяца.

### Переключатель режимов в боте: «normal/strict/calls»
**Триггер:** Появятся реальные сегменты юзеров с разными требованиями.
**Что:** Метод `XrayService.generate_config(user_id, mode)` — генерирует разные конфиги под сценарии.
**Почему отложено:** Усложнение UX без подтверждённого спроса.

### Реактивация ушедших юзеров через email
**Триггер:** База ушедших > 100 человек.
**Что:** Email-канал для тех у кого истекла подписка, скидка -40%.
**Почему отложено:** В TG-боте email юзеров обычно нет, нужен сбор. Telegram-рассылки эффективнее.
